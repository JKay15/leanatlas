#LeanAtlas Glossary (Must read for Codex)

This document defines recurring terms in the LeanAtlas workflow. The use of undefined terms is prohibited; if a new term must be introduced, define it here first.

## problem_slug
The unique identifier of a problem (Problem), corresponding to the directory name: `Problems/<problem_slug>/`.
Example: The slug of `Problems/fermat_little_theorem/` is `fermat_little_theorem`.

## OPERATOR mode
Default mode. Goal: Only complete the proof within `Problems/<problem_slug>/` without modifying the system code and contract.

## MAINTAINER mode
Maintainer mode. Goal: Modify the system (Toolbox/Incubator/tools/contracts/tests, etc.) to improve reusability, retrieval efficiency and robustness.
Entry method: There is a local `AGENTS.override.md` in the root directory of the warehouse (do not submit).

## small loop
Deterministic execution loop inside Codex:
Snapshot → Retrieval ladder → Attempt → Decide
The exit condition can only be: SUCCESS or TRIAGED.

## Snapshot
A "starting state snapshot" of an attempt:
- Workspace file content (hash)
- budget counters
- Retrieval trace (may be empty or partial)

## Retrieval ladder (retrieval ladder)
Candidate search order from "most reliable, cheapest" to "widest, most expensive".
Typical gradient: Toolbox → Seeds → Mathlib → External
(will be domain-driven in the future)

## Attempt
An attempt to "come up with a candidate proof and try to compile/verify it". Each Attempt must generate an AttemptLog line.

## Judge
Deterministic referee. Use only facts (patch_scope, budgets, signals, diagnostics) to make decisions:
- `CONTINUE`
- `TRIAGED`
- `SUCCESS`

## Advisor
Non-deterministic advisor (usually generated by the intelligence part of Codex itself).
Outputs hypotheses and suggested next steps, but does not override Judge.

## TRIAGED
Represents "exit the small loop, return the problem to the upstream (GPTPro or human)", and output structured evidence.
TRIAGED does not equal failure: it is a "compliant exit".

## PatchScope
File modification boundary determination. In OPERATOR mode, only a few files such as `Problems/<slug>/Proof.lean` are allowed to be modified.
Out-of-bounds modifications must be TRIAGED.

## RunReport / RetrievalTrace / AttemptLog
Forced products for each run:
- RunReport: Overview of this run (final status, key indicators, links)
- RetrievalTrace: retrieval process (steps, hits, budget consumption)
- AttemptLog: Facts + referees + staff records of each attempt (jsonl)

## Toolbox / Incubator / Seeds
- Toolbox: Official reusable tool library (requires deduplication access control and reuse evidence)
- Incubator: incubation area
- Seeds: candidate tools that can be searched and hit in the incubation area

## lake build
The official build command for Lean 4: Compile the current project (generate .olean, etc.), used to verify the "truth of the local environment".

## lake test
Run the project's configured testDriver (test driver). It should be used as the standard entrance for "running workflow tests (e2e/scenario/soak + core gate)" in LeanAtlas.

## lake lint
Runs the project's configured lintDriver (static checking/discipline checking). LeanAtlas should include: contract/Schema/product specification, test pollution scan, structure/dependency discipline, etc.


## MCP(Model Context Protocol)
A Host/Client/Server tool calling protocol, Codex can call external tools (retrieval, LSP, database, etc.) through MCP.
In LeanAtlas: MCP can only be accelerated, not authoritative; there must be a downgrade path (see `docs/contracts/MCP_ADAPTER_CONTRACT.md` for details).

## lean-lsp-mcp
An external MCP server: Provides the capabilities of Lean Language Server (diagnostics, code actions, local search, etc.) to the agent in the form of tools.
In LeanAtlas: As one of the executors of the proof-loop, but cannot become a single point dependency.

## Automation (background automation tasks)
Codex App/CLI can run background tasks on a scheduled basis for "unattended" quality access control and library maintenance.
LeanAtlas's automation must be split into: deterministic pre-step + (optional) Advisor step, and incorporated into TDD (see `docs/agents/AUTOMATIONS.md` for details).

## worktree (background workspace)
A separate working directory for Git worktree. Codex App Automations are usually run in a separate worktree in Git projects to avoid polluting the main workspace.

## MSC2020
Mathematics Subject Classification 2020: Mathematics field classification system. LeanAtlas uses it as domain ontology for retrieval pruning, GC bucketing, etc.
